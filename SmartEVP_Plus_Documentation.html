01 System Overview

One call. Five systems.
Zero delay.

SmartEVP+ is not a single device. It is a chain of five tightly integrated intelligence systems that hand off seamlessly — from the moment a citizen calls for help to the moment a doctor receives a complete pre-arrival patient report.

02 Subsystem Breakdown

Five systems,
one mission.

Each subsystem is independently buildable but tightly coupled through MQTT and REST APIs. Here is the complete breakdown of what every component does, how it works, and what it produces.

🎙
Subsystem 01 — Emergency Intake
AI Voice Agent
When the emergency number is called, VibeVoice-ASR transcribes the conversation in real time. A Python agent parses the transcript to extract structured data — location, symptom keywords, caller language, and urgency level.
  • Twilio handles inbound call routing to a Python webhook
  • VibeVoice-ASR-7B processes the audio stream for transcription with speaker diarization and timestamps
  • Hotwords configured: "chest pain", "breathing", "unconscious", "accident", "bleeding", "stroke" and more
  • Gemma 4 E4B parses transcript → outputs structured JSON: { location, severity, symptoms[] }
  • Fallback: Manual SMS input if voice fails indoors
🗺
Subsystem 02 — Smart Dispatch
Nearest Ambulance Engine
The dispatch engine receives the structured emergency JSON, looks up active ambulances from a simple in-memory store (GPS positions updated via MQTT), computes distances, and allocates the nearest available unit.
  • Haversine formula used for real-time distance computation between caller and ambulances
  • Ambulance status tracked: AVAILABLE / EN_ROUTE / ON_SCENE
  • Twilio SMS fired to driver with patient location, case ID, and urgency
  • If no movement detected within 90 seconds — second SMS + escalation alert triggered
  • Controller dashboard receives real-time dispatch event via WebSocket
🚦
Subsystem 03 — Traffic Preemption
SmartEVP Core
The original SmartEVP system. ESP32 on the toy ambulance publishes GPS coordinates every second via WiFi to the MQTT broker on the Raspberry Pi. The Pi monitors proximity and triggers GPIO to flip LEDs green.
  • ESP32-WROOM-32 reads NEO-6M GPS at 1Hz, publishes to topic: evp/gps/<vehicle_id>
  • Pi subscribes, computes distance to virtual intersection coordinates
  • At <500m threshold: publishes to evp/signal/cmd → GPIO pin 18 triggers relay
  • ECC-256 signature on each GPS payload for authentication (OpenSSL)
  • Fallback: Coordinate replay from pre-recorded JSON route if GPS signal is weak indoors
  • Latency target: <5s end-to-end (achieved 3.2s in testing)
🧠
Subsystem 04 — Edge AI
En-route Medical Intelligence
While the ambulance is en route, the Raspberry Pi listens to the ambulance microphone audio stream (nurse/EMT narrating patient status). VibeVoice-ASR transcribes it. Gemma 4 E4B converts raw transcript into a clean, doctor-ready medical brief.
  • Audio input via USB microphone attached to Raspberry Pi (demo: pre-recorded clip played)
  • VibeVoice-ASR processes audio — produces timestamped, speaker-diarized transcript
  • Gemma 4 E4B (4.5B active params, Ollama on Pi) generates structured medical brief
  • Output fields: Chief complaint, vitals, allergies, medications, suspected diagnosis, resources needed, estimated arrival
  • Report published to MQTT topic: evp/medical/report → forwarded to hospital dashboard
  • Fully offline capable — no cloud API needed for medical processing
🏥
Subsystem 05 — Hospital Readiness
Pre-Arrival Dashboard
The hospital receives the complete patient brief before the ambulance arrives. The dashboard auto-populates with all medical data, shows a live ETA countdown, and lists required resources — so the medical team is ready the moment the patient enters the building.
  • React dashboard connects to Flask backend via WebSocket for live updates
  • Patient card populates automatically — zero manual data entry
  • ETA countdown computed from live GPS position and estimated route distance
  • Resource flags auto-set: Stretcher, Defib, ICU bed, Cardiologist, OR
  • PDF export of pre-arrival report available for physical handoff
  • Multi-case view for hospitals handling multiple simultaneous emergencies

04 AI Models

Two models.
All the intelligence.

The entire AI layer is built on two open-source models — one for voice understanding, one for medical reasoning. Both run locally where possible, with cloud API as a fallback for the hackathon demo.

🎙 Microsoft
VibeVoice-ASR-7B
Hugging Face: microsoft/VibeVoice-ASR
A unified speech-to-text model that processes up to 60 minutes of audio in a single pass. It jointly performs transcription, speaker diarization, and timestamping — producing a rich, structured output of who said what and when. Multilingual, supporting 50+ languages including Hindi, Kannada, and other Indian languages.
Where it runs: Laptop / cloud API for hackathon demo. Raspberry Pi is too limited for 7B inference — use Hugging Face Inference API or a laptop running it locally.
Key capability used: Long-form single-pass ASR with speaker diarization for the ambulance en-route audio capture.
Customized hotwords: Pre-register medical terms — "myocardial infarction", "GCS score", "BP 90/60" — to boost accuracy on emergency audio.
Output: Structured JSON with speaker labels, timestamps, and full transcription → fed into Gemma 4 for report generation.
vLLM support: For production, VibeVoice-ASR supports vLLM inference for significantly faster throughput.
🤖 Google DeepMind
Gemma 4 E4B
4.5B active parameters — Runs on Raspberry Pi via Ollama
Gemma 4's most efficient dense model, with 4.5B effective parameters and a 128K context window. It runs on consumer hardware via Ollama — including Raspberry Pi 4 with 4GB+ RAM. Used for two tasks: parsing emergency intake transcripts into structured JSON, and converting raw ambulance audio transcripts into clean, doctor-ready medical briefs.
Where it runs: Ollama on Raspberry Pi 4 (4GB RAM). Quantized to Q4_K_M for efficient inference. Inference time: 15–45 seconds per report — acceptable for the demo.
Task 1 — Intake parsing: Receives raw transcription of emergency call → outputs structured JSON: { "location": "...", "severity": "HIGH", "symptoms": [...], "caller_language": "Kannada" }
Task 2 — Medical brief: Receives VibeVoice-ASR transcript of en-route nurse audio → outputs formatted medical brief with Chief Complaint, Vitals, Suspected Diagnosis, Resources Required, ETA.
Architecture: Hybrid attention (sliding window 512 + full global attention), 128K context window, multilingual, native function calling support.
Privacy angle: Runs fully offline on the Pi. Patient audio and medical data never leave the device. HIPAA-compliant by design.
ℹ
For the hackathon demo, run VibeVoice-ASR via its online Playground API or via the Hugging Face Transformers library on a laptop. Run Gemma 4 E4B on the Raspberry Pi via Ollama. This way the Pi stays cool and you get fast ASR without taxing the hardware.

07 Demo Execution

The 90-second
money shot.

This is how you present SmartEVP+ during the hackathon. Every second is scripted. Every judge moment is pre-planned. The demo tells a story — and judges feel every beat of it.

🎯
Setup before demo: Pi is running and connected to hotspot. MQTT broker active. Dashboard open on laptop (full screen, dark mode). ESP32 powered and publishing GPS coords. LEDs on table, clearly visible to all judges. Twilio number ready on your phone.
T = 0s — The Call
Judge calls the Twilio number live
You hand your phone to a judge (or call it yourself). The AI voice agent answers instantly — no ringing, no waiting. It says: "Emergency response system active. Please describe your emergency." The judge says: "Heart attack patient near Koramangala, BTM Layout junction."
T = 8s — Intake Complete
Dashboard lights up. SMS lands on the table.
The controller dashboard shows a new case card appearing live. Severity: HIGH. Location: BTM Layout, Bengaluru. The driver phone sitting on the table buzzes — judges see the SMS notification land in real time. It reads: "CASE #0041 — DISPATCH IMMEDIATE. Patient: Cardiac event, BTM Layout junction. Respond now."
T = 15s — Ambulance Moves
The toy ambulance is pushed. The map dot moves.
You push the toy ambulance slowly across the table. On the big screen, everyone watches the GPS dot animate across the Bengaluru map — live, smooth, real. The dashboard shows speed, bearing, ETA countdown. The ambulance is "en route."
T = 35s — THE MOMENT
The ambulance crosses 500m. The lights flip.
The GPS coordinate crosses the threshold. The Pi detects it. MQTT fires. GPIO activates. The three LEDs on the breadboard — previously red — flip to green, live, in front of everyone. The dashboard simultaneously shows "SIGNAL PREEMPTION ACTIVE — Intersection 01 — GREEN."
⬜⬜🟢 This is your silence moment. Say nothing. Let it breathe for 3 seconds.
T = 50s — En-Route Intelligence
Nurse audio plays. Gemma 4 transcribes and understands.
You play a 15-second pre-recorded audio clip of a nurse narrating: "Male patient, 58 years old, complaining of chest pain radiating to left arm, BP 90 over 60, HR irregular at 112, SpO2 at 94%. No known allergies. On Aspirin 75mg daily." VibeVoice-ASR transcribes it live on screen. Gemma 4 processes it silently.
T = 70s — Hospital Knows Everything
The hospital dashboard auto-fills. Before the ambulance arrives.
Switch the screen to the Hospital Dashboard tab. The medical brief has auto-populated: Chief Complaint: Acute Chest Pain. Suspected: STEMI. Resources Required: Cardiologist, Defib, ICU bed. ETA: 4 minutes. The patient hasn't arrived — but the hospital is ready.
Then say: "By the time those doors open, the doctor already knows everything."
💚
Indoor GPS fallback: If GPS struggles inside the venue, pre-load a 60-point JSON coordinate sequence that traces a route toward the virtual intersection. A Python script replays this at 1Hz, publishing to MQTT exactly as the ESP32 would. Judges see identical behavior. Keep this as your silent backup — activate if GPS fix doesn't come within 60 seconds of powering up.
